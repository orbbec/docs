132/*
133 Standards for using the tracing Macros:
134
135 Parameter validation:
136 All parameters that requires validation should be checked at the start of a function.
137 Parameters should be checked in the order they are declared.
138 One check should be used per parameter.
139 Use RETURN*_IF_HANDLE_INVALID for handles created with handle.h
140 Use RETURN*_IF_ARG for other types
141
142 Calling other functions:
143 If the callee function returns a k4a_result_t, wrap the call in TRACE_CALL.
144 TRACE_CALL will pass through the result, and trace the error information if one occurs
145 If the callee function returns another type, K4A_RESULT_FROM_BOOL will convert a boolean success expression to a
146 a k4a_result_t, and trace the error information if one occurs
147
148 Checking an error condition expression:
149 When checking a condition that indicates a success or failure, wrap the expression in TRACE_BOOL and use the
150 k4a_result_t result.
151
152 Checking for failures:
153 The preferred pattern is to initialize a result variable as K4A_RESULT_SUCCEEDED, then update that variable with
154 each call or operation in the function. If a subsequent operation shouldn't be performed after a failure,
155 wrap it in an "if (K4A_SUCCEEDED(result))" block
156
157 Cleaning up from errors:
158 Generally, if a function returns K4A_RESULT_FAILED, it should not have any side effects on persistent state.
159 This means cleaning up any memory that was allocated before the failure, or undo-ing any operations that were
160 partially completed.
161
162 Functions should check for failure with a K4A_FAILED(result) check, and perform all the needed cleanup. If there
163 the failing function is a "create" function, you may wish to call the respective "destroy" at this point to avoid
164 duplicating cleanup logic.
165*/
